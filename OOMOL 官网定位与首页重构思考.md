# OOMOL 官网定位与首页重构思考

## 背景

当前官网的主要问题，不是信息太少，而是主价值没有被压缩成一个足够清晰的用户认知。

过去的表达更偏向：

- 我们做了工作流
- 我们支持 AI tool
- 我们支持 MCP / API / 自动化
- 我们有 Studio / Cloud / Community / SDK

这些信息都是真的，但它们更像能力列表，不像一个明确的价值承诺。结果是开发者看完之后，知道 OOMOL 做了很多事，却不一定立刻明白：

`这个产品到底为什么值得我用？`

## 核心判断

### 1. 不应该继续把 OOMOL 主讲成“工作流平台”

原因不是工作流不重要，而是“工作流平台”这个心智已经被既有品类占住了。

- 如果用户按工作流平台理解 OOMOL，会自然拿它和 `n8n` 比生态、低代码成熟度、连接器丰富度
- 如果用户按 AI workflow / AI app builder 理解 OOMOL，会自然拿它和 `Dify` 比 AI 场景模板、agent 能力、RAG 方案完整度

这不是 OOMOL 最强的位置。

工作流对 OOMOL 来说更适合作为：

- 能力组合方式
- 交付过程中的编排方式
- 函数能力向更大服务演进的组织形式

而不是首页第一定义。

### 2. 也不应该继续把 OOMOL 主讲成“AI tool 平台”

“AI tool” 这个词太泛。

它既可能指：

- Chat 里调用的小工具
- MCP server
- skill
- agent 的工具库
- 低代码 AI 应用

如果继续用这个词做一级定位，会让 OOMOL 的真正优势被淹没。

用户会很难区分：

- OOMOL 和 MCP 工具生成器有什么本质区别
- OOMOL 和内置脚本 / skill 有什么本质区别
- OOMOL 到底是开发工具，还是工具分发平台

### 3. OOMOL 最强的位置，是“开发者友好的 Function-to-Service”

整个产品最有辨识度的不是“也能编排工作流”，而是：

- 在真实 coding 环境里工作
- 基于 VSCode 和容器
- Agent 可以帮助生成函数单元
- 本地就能调试和验证
- 同一份能力可以直接发布为服务

所以最应该占住的心智不是 workflow，不是 AI tool，而是：

`把函数做出来，把服务发出去。`

或者更完整一点：

`在真实 coding 环境里生成函数，本地验证后，直接发布成服务。`

## 产品定位结论

### 推荐定位

OOMOL 应该被定义为：

`给开发者的 AI-native Function-to-Service 平台`

### 面向人群

- 会写代码的开发者
- 希望把能力快速交付出去的团队
- 不想先搭后端、运维、服务框架，再去上线一个函数的人

### 核心价值

- 不用为了可视化或发布去学习一套新 DSL
- 继续在真实工程环境里工作
- 让 Agent 帮助生成和补全函数单元
- 本地调试与云端交付路径打通
- 同一份能力直接发布成 API、MCP 工具或自动化任务

### 一句话主张

当前最合适的首页主张仍然是：

`写一个函数，上线一个服务`

问题不在 slogan，本质问题在于过去首页后续内容没有继续把这句话讲透。

## 官网表达原则

### 1. 首页前三屏只围绕一条主线

首页不应该同时承担太多任务。前三屏应该只讲清楚这一条链路：

`函数生成 -> 本地验证 -> 服务交付`

### 2. 术语顺序要按用户理解来，不要按产品内部结构来

首页前两屏应该优先使用这些词：

- 函数
- 代码
- 本地调试
- 服务
- API / MCP / Automation

而不是一上来就使用：

- Block
- Workflow
- Package
- Hub
- 生态

更合适的术语关系是：

- `Function`：用户第一认知
- `Block`：产品内部实现或可复用单元
- `Workflow`：函数之间的组合方式
- `API / MCP / Automation`：交付方式

### 3. MCP 应该作为交付方式，不应该作为产品定义

MCP 很重要，但它更适合作为：

- 一种发布目标
- 一种调用协议
- 一种被 Agent 使用的交付形式

而不是 OOMOL 的一级定位。

### 4. 工作流应该降级为能力组织方式

OOMOL 不需要否认 workflow，而是应该重新安排它的层级：

- 首页第一主语是“函数能力交付”
- workflow 在第二层负责解释“如何组合”
- 发布页和产品页再详细解释 workflow 的价值

### 5. 社区要从“展示社区”改成“沉淀团队资产”

如果首页后半段继续讲“社区发现、分享、灵感”，会让整个页面的重心又偏回内容平台或工具社区。

更强的表达应该是：

- 做好的函数能力会沉淀为团队资产
- 团队可以安装、调用、继续组合
- 一次开发，不是一锤子买卖，而是持续长成更多服务

## 首页结构建议

### 第一屏：一句话讲主价值

- 标题：`写一个函数，上线一个服务`
- 副标题：强调真实 coding 环境、Agent 生成、本地验证、服务发布
- 辅助标签：强调不是新 DSL、容器本地调试、同一份能力直接发成服务
- 视觉截图：直接叠加 `Code -> Validate -> Ship`

### 第二屏：解释为什么 OOMOL 不是普通 workflow 工具

需要回答的问题是：

`为什么我不直接用一个普通 workflow 平台或者 AI IDE？`

重点应放在：

- 真实 coding 环境
- 不迁就陌生框架
- Agent 生成函数单元

### 第三屏：明确三种交付方式

这一屏应该明确说明：

- 发成 API
- 发成 MCP Tool
- 发成 Automation

而不是把它们混在一段描述里。

### 第四屏：从社区感转向资产感

重点表达：

- 已有函数能力可安装
- 团队可复用
- 能继续组合成更大的服务

## 已在项目中落地的改动

### 1. Hero 主叙事已重写

已将首页首屏文案收紧为：

- 强调真实 coding 环境
- 弱化 workflow 作为一级概念
- 强化函数单元生成与服务交付

### 2. Hero 增加定位标签与视觉注释

已增加：

- `AI-Native Function-to-Service` kicker
- 三个价值 pill
- 截图上的三张视觉注释卡片
- `Code -> Validate -> Ship` 交付链

### 3. 三步区块已重构为更可扫读的结构

已将原本偏长文的说明改成：

- 步骤标签
- 核心说明
- bullet 列表
- 第三步的三张交付卡片

### 4. CommunityShare 已改为团队资产视角

已从“发布后继续复用”进一步重构为：

- 团队资产沉淀
- 可安装能力目录
- 安装、调用、继续组合

## 后续建议

### 建议 1：继续减少首页上的产品生态感

首页的目标是让人快速理解价值，不是完整介绍产品全景。

后续可以继续检查以下内容是否还会稀释焦点：

- 过早出现产品生态
- 过早出现过多产品名
- 过早出现过多场景卡片

### 建议 2：用更多“开发者收益”替代“平台能力”

优先表达：

- 不学新 DSL
- 保留真实工程习惯
- 本地先跑通
- 不重搭后端
- 不自己管运维

少表达：

- 我们支持很多模式
- 我们支持很多产品形态
- 我们的生态很完整

### 建议 3：后续所有产品页都应服从首页定位

后续如果继续改 `Studio`、`Cloud`、`SDK` 等页面，建议统一遵循这条主线：

- `Studio` 负责讲：为什么这里适合生成和验证函数能力
- `Cloud` 负责讲：为什么这里适合把函数能力服务化
- `Community / Hub` 负责讲：为什么这些能力可以被沉淀、安装和复用

## 最终结论

OOMOL 的官网不应该再试图证明：

- 我们也能做 workflow
- 我们也能做 AI tool
- 我们也有 MCP

而应该让开发者立刻理解：

`我可以在一个真实可控的 coding 环境里，让 AI 帮我把函数做出来，然后直接把这份能力发成服务。`

这就是 OOMOL 当前最值得占住的心智。
