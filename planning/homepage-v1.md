# OOMOL 官网首页文档 V1

## 1. 文档目的

这份文档用于统一 OOMOL 官网首页的第一版优化方向。

当前阶段的目标不是先强调和 Composio 的差异，而是先让访客快速理解：

- OOMOL 属于哪一类产品
- OOMOL 可以帮用户完成什么
- 用户为什么应该从首页继续往下看
- 用户应该从哪个入口开始体验

这份文档聚焦官网首页，不展开到所有二级页面。

## 2. 当前阶段的首页目标

首页的核心任务有三个：

1. 建立类别认知
   让用户在很短时间内理解，OOMOL 是一个为 Agent、Codex、Claude Code 这类环境提供工具连接、能力封装、能力运行与交付的平台。

2. 建立使用路径认知
   让用户理解，OOMOL 不只是“接一个工具”，而是能从接入能力、组合能力，到运行、交付、被 Agent 使用，形成一条连续路径。

3. 建立开始动作
   让用户清楚知道自己下一步该做什么，而不是在多个产品名之间犹豫。

## 3. 当前阶段不作为首页主叙事的内容

以下内容不是不要讲，而是不作为首页最前面的主叙事：

- 代码不暴露
- 开源还是闭源
- 与 Composio 的差异化对比
- 更深层的平台架构细节

这些内容更适合放到后面的 Cloud、Trust、FAQ、Comparison 页面中承接。

## 4. 首页应该先解决的认知问题

首页访问者进入时，通常会先问这几个问题：

1. 你们到底是什么产品？
2. 你们是给谁用的？
3. 我能不能用你们来给 Codex / Agent 接工具和能力？
4. 我只能接现成 SaaS，还是也能接自己的 API、内部系统、自定义逻辑？
5. 我应该先装 CLI，还是先看 Studio，还是直接接 API？

首页的结构，必须围绕这几个问题来组织。

## 5. 首页建议定位

### 5.1 一句话定位

建议首页先采用偏赛道共识的表达，不急着讲复杂差异：

`为 Codex 和 AI Agent 构建、连接并交付可用能力`

### 5.2 一段补充说明

建议用一段更完整的话把产品范围讲清楚：

OOMOL 帮助开发者连接 SaaS 授权和 API Key，封装自定义 API、内部系统与业务逻辑，把这些能力组合成可被 CLI、API、MCP 和 Agent 使用的运行入口。

### 5.3 当前阶段的定位原则

- 先让用户知道你们属于这条赛道
- 再让用户知道你们不只支持现成连接
- 最后再引出 Studio、Cloud、skill 模式这些更强的能力

## 6. 首页目标用户

首页主要面向三类用户：

### 6.1 Agent / Codex 使用者

这类用户想要的是：

- 让 Codex、Claude Code 等工具能直接调用能力
- 尽快开始使用已有能力
- 不想先接一堆底层协议

### 6.2 开发者 / 工程团队

这类用户想要的是：

- 接入 SaaS auth 或 API key
- 封装自己的 API 或内部逻辑
- 在真实编码环境里做开发和验证
- 最后能交付给自己、团队或客户使用

### 6.3 平台型或产品型团队

这类用户想要的是：

- 一套能力，多种使用入口
- 不同入口共享同一份实现
- 从内部使用逐步走向外部交付

## 7. 首页信息架构建议

首页建议采用下面的顺序。

### 7.1 第一屏：先讲产品类别和价值

目标：

- 用最短时间说清楚“你们是什么”
- 不先陷入产品名和产品结构

要讲清楚的点：

- 给 Codex / AI Agent 提供能力连接与运行
- 支持 SaaS、API key、自定义 API、内部逻辑
- 最终可以被 CLI、API、MCP 使用

建议内容结构：

- 标题
- 一句话解释
- 两个主按钮
- 一个简单的能力示意图或短视频

建议按钮：

- 主按钮：`Try with oo-cli`
- 次按钮：`See how it works`

### 7.2 第二屏：讲用户能接入哪些能力

目标：

- 让用户快速建立“能力范围”的认知

建议分三类能力展示：

- SaaS tools
- API-based capabilities
- Custom internal logic

建议每类都用短句说明：

- 连接现成 SaaS
- 接入带 auth 或 API key 的服务
- 封装自己的接口、脚本和业务逻辑

这一屏的任务是让用户理解：

OOMOL 不是只能接现成工具，也不是只能做自定义工作流，而是两边都能承接。

### 7.3 第三屏：讲最短使用路径

目标：

- 不先讲所有产品
- 先讲用户如何开始

建议采用双路径结构：

- 路径 A：先用已有能力
- 路径 B：自己构建并发布能力

可用表达：

#### 路径 A

安装 `oo-cli`，搜索、查看并运行已发布能力。

#### 路径 B

在 `OOMOL Studio` 中生成和组合能力，在本地验证，再通过 `Cloud` 持续运行与交付。

这一屏比现在更重要，因为它比“产品并列展示”更接近用户的真实决策流程。

### 7.4 第四屏：讲交付与使用入口

目标：

- 让用户理解“一份能力，多种入口”

建议展示四种入口：

- CLI
- API
- MCP
- GUI / App surface

这一屏不要讲太深的技术细节，只讲结果：

- 给 Codex 用
- 给自有应用用
- 给 Agent framework 用
- 给终端用户用

### 7.5 第五屏：讲开发者为什么会选择 OOMOL

目标：

- 承接更强的产品优势

建议讲以下几个点：

- 真实编码环境，不是新的 DSL
- 支持组合和扩展，不只消费现成工具
- 本地验证后再交付
- 同一份实现复用到多个入口
- 支持 skill 模式扩展

这部分是首页里真正建立“你们更强”的位置，但不需要先和竞品正面比较。

### 7.6 第六屏：讲 Cloud 的承接价值

目标：

- 说明为什么需要 Cloud

这里再放入下面这些内容会更自然：

- 统一运行时
- Secrets / 配置管理
- 使用数据
- 订阅与交付
- 代码不暴露

这里尤其适合讲“代码不暴露”。
因为它在这个位置是一个交付层优势，而不是整个产品的唯一理由。

### 7.7 第七屏：结尾 CTA

目标：

- 让用户不带困惑地进入下一步

建议只保留两个动作：

- `Start with oo-cli`
- `Build your first capability`

如果需要企业转化按钮，可增加：

- `Talk to us`

## 8. 首页文案方向草稿

以下是首页文案方向草稿，不是最终定稿。

### 8.1 Hero 草稿

标题：

`为 Codex 和 AI Agent 接入可用能力`

副标题：

`连接 SaaS、API key 和自定义逻辑，把能力封装成可被 CLI、API、MCP 和 Agent 使用的运行入口。`

补充说明：

`先用已有能力，也可以在 OOMOL Studio 里生成、组合和扩展你自己的能力，再通过 Cloud 持续运行与交付。`

按钮：

- `Try with oo-cli`
- `See how it works`

### 8.2 能力范围草稿

区块标题：

`不只是接现成工具，也能封装你自己的能力`

三列文案：

- `Connect SaaS`
  连接常见 SaaS 服务和已有接口能力

- `Bring APIs`
  接入带 auth 或 API key 的 API 服务

- `Build Custom Logic`
  组合你自己的 API、脚本、内部系统和业务逻辑

### 8.3 双路径草稿

区块标题：

`从使用到构建，一条连续路径`

路径 A 标题：

`先使用已有能力`

路径 A 描述：

`通过 oo-cli 在 Codex、Claude Code 或终端中搜索、查看并运行已发布能力。`

路径 B 标题：

`再构建自己的能力`

路径 B 描述：

`在 OOMOL Studio 中生成与组合能力，本地验证后，通过 Cloud 运行并交付给自己、团队或客户。`

### 8.4 使用入口草稿

区块标题：

`同一份能力，多种使用入口`

说明文案：

`一个能力可以先给开发者在 CLI 中使用，也可以接成 API、封装成 MCP，或者继续承载到图形界面中。`

### 8.5 开发者价值草稿

区块标题：

`开发者不该为了交付能力而切换一整套工作方式`

要点文案：

- 不引入新的平台 DSL
- 在真实编码环境中开发和验证
- 能组合和扩展，而不是只消费预设能力
- 一份实现，复用到多个入口
- skill 模式可扩展

### 8.6 Cloud 承接草稿

区块标题：

`当能力需要持续运行和交付时，Cloud 来承接`

说明文案：

`把运行时、配置、Secrets、订阅和使用数据统一放到 Cloud 中管理。需要时，能力在云端运行，用户只使用入口，不必接触底层实现。`

## 9. 首页 CTA 策略建议

当前首页不建议继续把“下载桌面版”作为唯一最强主动作。

建议的 CTA 层级如下：

### 一级 CTA

- `Start with oo-cli`

这是最像“立即开始”的动作，也最符合 Agent / Codex 用户场景。

### 二级 CTA

- `See how it works`

适合还没有准备安装的人。

### 三级 CTA

- `Build with Studio`
- `Open Cloud Console`

这些动作更适合已经理解产品的用户。

## 10. 导航建议

首页优化后，导航建议围绕用户任务，而不是完全围绕产品名。

当前阶段可保持产品名不变，但建议强调进入逻辑。

建议导航结构：

- Product
- Docs
- Pricing
- Blog / Updates
- Get Started

其中 `Product` 下再分：

- oo-cli
- Studio
- Cloud
- App Surfaces

`OOMOL AI` 如果继续保留在主导航中，建议明确它是产品化使用入口，而不是平台主叙事中心。

## 11. 首页优化的阶段性原则

### 11.1 第一阶段

先学会让用户正确理解你们属于哪一类产品。

### 11.2 第二阶段

再让用户意识到你们不只是连接器平台，还能做自定义能力生产与交付。

### 11.3 第三阶段

最后再补充：

- 为什么你们比竞品更灵活
- 为什么 Studio + Cloud 是优势
- 为什么 skill 模式更适合生态扩展

## 12. 这份文档之后可以继续拆出的工作项

这份首页文档确定后，下一步可以继续拆成三份更具体的执行稿：

1. 首页线框文档
   把每一屏的组件结构和视觉层级拆出来

2. 首页正式文案稿
   直接写成可以进入设计或开发的中英文文案

3. 首页改版任务清单
   明确哪些模块保留、哪些模块重写、哪些模块下沉到二级页

## 13. 当前建议结论

当前首页最重要的不是先讲差异化，而是先让用户在最短时间内理解：

OOMOL 是一个面向 Codex 和 AI Agent 的能力连接、构建、运行与交付平台。

首页应该先讲类别认知、能力范围和使用路径，再逐步引出 Studio、Cloud、skill 模式，以及代码不暴露等后续优势。
