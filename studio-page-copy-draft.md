## 第一部分：为什么需要 OOMOL Studio

### 标题

做工具，不该卡在环境、上手和扩展上

### 说明

很多平台可以帮助你把流程连接起来，但当项目进入真实开发阶段，问题往往才刚刚开始。如何更快完成第一版，如何减少环境带来的反复调试，如何在需要时顺畅接手并继续扩展，才是真正决定效率的关键。

### 卡片 1

#### 标题

从想法到第一版，往往比预期更慢

#### 文案

需求一旦进入真实项目，开发者不仅要写代码，还要处理结构组织、输入输出、依赖配置和功能衔接。很多时间并不是花在实现能力本身，而是花在把零散想法真正落成可运行结果上。

### 卡片 2

#### 标题

本地开发和最终部署，常常不是同一种状态

#### 文案

如果开发环境、调试环境和线上运行环境缺少一致性，问题就会在交付阶段集中出现。依赖差异、版本变化和运行方式不一致，都会让本来简单的事情变得复杂。

### 卡片 3

#### 标题

当需要深入修改时，很多工具就不再顺手

#### 文案

真实项目总会遇到需要手动调整的时刻。无论是补代码、查日志、调整参数，还是接入第三方库，如果平台无法支持开发者按工程方式继续推进，效率很快就会被拉低。

## 第二部分：OOMOL Studio 提供了什么

### 标题

一套面向真实生产的工具开发环境

### 说明

OOMOL Studio 不只是一个可视化编排工具，它更像一个围绕工具开发设计的工作台。AI 辅助、本地环境、代码编辑、依赖扩展和最终交付都在同一套体系内完成，让开发过程更完整，也更可控。

### 能力 1

#### 标题

内置 AI Agent，帮助你更快完成第一版

#### 文案

OOMOL Studio 内置 AI Agent，可以直接参与功能开发。你可以用自然语言描述需求，让它帮助你生成代码、补全逻辑、搭建结构，加快从想法到可运行结果的过程。

它的价值不只是节省输入时间，更在于帮助你更快进入可验证、可修改、可继续推进的开发状态，让第一步不再成为负担。

### 能力 2

#### 标题

本地容器环境与版本管理，减少环境带来的不确定性

#### 文案

OOMOL Studio 在本地提供稳定的容器化开发环境，让调试过程尽可能接近最终部署状态。这样做不是为了增加一层技术概念，而是为了减少环境差异带来的反复排查和重复劳动。

配合版本管理能力，项目中的依赖、改动和运行结果都更容易被追踪和复现，让开发、验证和交付形成更清晰的连续过程。

### 能力 3

#### 标题

基于 VS Code 的设计，方便随时接手和深入调整

#### 文案

OOMOL Studio 基于开发者熟悉的 VS Code 逻辑设计。当 AI 不能完全满足需求时，你可以立即进入代码、配置、日志和项目结构，继续以自己熟悉的方式完成修改和优化。

这意味着 AI 与人工并不是割裂的两套流程，而是在同一个环境中自然衔接。该自动的时候自动，该手动的时候手动，整个过程始终保持顺畅。

### 能力 4

#### 标题

支持 Python 与 JavaScript 生态，扩展方式更符合开发习惯

#### 文案

真实开发离不开成熟的开源生态。无论是处理 PDF、进行数学计算，还是连接外部服务，OOMOL Studio 都支持按社区习惯引入 Python 和 JavaScript 的第三方库，让项目拥有更完整的扩展能力。

这不仅提升了开发效率，也意味着你不必受限于平台预设能力，而是可以把已有技术积累直接带入自己的工具开发流程。

## 第三部分：如何理解 OOMOL Studio 的工作方式

### 标题

不是只把流程连起来，而是把能力沉淀下来

### 文案

在 OOMOL Studio 里，你可以把单个功能实现为可复用的模块，再把多个模块组合成完整流程，并将已经验证过的流程继续封装，形成新的可复用能力。

这种方式带来的价值，不只是把当前任务完成，而是让项目中的成果能够持续复用、不断扩展，并逐步沉淀成一套真正可交付的工具体系。
